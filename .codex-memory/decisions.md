# Decisions
## 2026-07-03 - Split Image Assets Switches The Active Canonical Line To Planning-First Generation Routing
- Decision: Replace the prior installer-UX-first active milestone with a planning-first route-control milestone that adds whole-image planning, `plan_manifest`, a formal `generation_routing` gate, explicit object routes, generation capability truthfulness, and `generated-reconstruction` delivery semantics.
- Rationale: The user clarified that the most important current problem is not setup friction but wasted extraction effort and weak outcomes on objects that should have been diverted earlier to reconstruction, rebuild, or constrained generation. The package already has strong packaging and honesty gates; it now needs earlier route control and clearer generated-delivery semantics.
- Impact: `docs/superpowers/split-image-assets/design.md` and `implementation-plan.md` now treat planning-first route control as the active authority. The next code pass should wire that design into package docs, preflight capability reporting, validator rules, review flows, and tests.
- Related files: `docs/superpowers/split-image-assets/design.md`, `docs/superpowers/split-image-assets/implementation-plan.md`, `.codex-memory/project-state.md`, `.codex-memory/session-log.md`

## 2026-07-03 - Split Image Assets Planning-First Rollout Must Stay Compatibility-First
- Decision: Add explicit rollout compatibility rules to the canonical planning-first docs: map `planned_route` to current `decision_routing`, stage `plan_manifest` enforcement, define generation provider classes, and treat new generated/accepted statuses as coordinated contract migrations rather than isolated field additions.
- Rationale: A deep doc review showed that the planning-first direction was right but still dangerous to implement directly. Without explicit migration tables, developers would likely create two routing taxonomies, break current pass gates, or globally hard-fail legacy packages too early.
- Impact: `design.md` now contains a compatibility strategy, route mapping table, delivery-state compatibility matrix, validator rollout stages, and provider-class contract. `implementation-plan.md` now starts with a Phase 0 alignment pass and requires non-generated legacy-compatible paths to stay green until an explicit migration phase says otherwise.
- Related files: `docs/superpowers/split-image-assets/design.md`, `docs/superpowers/split-image-assets/implementation-plan.md`, `.codex-memory/session-log.md`

## 2026-07-03 - Split Image Assets Uses Staged Generated-Route Enforcement
- Decision: Implement generated-route enforcement as a staged validator path: require `plan_manifest.json` and generated evidence only for generated-route objects first, while preserving legacy-compatible non-generated packages and legacy-compatible approximate pass behavior during migration.
- Rationale: The new milestone needed real generated-reconstruction truthfulness without breaking the stable non-generated package baseline. Generated-route hard gates are high value and low ambiguity; global hard-failure on every package would have created unnecessary migration churn.
- Impact: `check_extraction_environment.py` now reports generation capability separately, `init_asset_package.py` now writes `plan_manifest.json`, the shared contract includes `generation_routing`, generated delivery classes, and new reuse statuses, and `validate_asset_package.py` now enforces plan/evidence requirements for generated-route objects while leaving non-generated legacy-compatible packages valid.
- Related files: `split-image-assets/scripts/check_extraction_environment.py`, `split-image-assets/scripts/init_asset_package.py`, `split-image-assets/scripts/split_image_assets_contract.py`, `split-image-assets/scripts/record_quality_review.py`, `split-image-assets/scripts/validate_asset_package.py`, `split-image-assets/tests/test_skill_package.py`

## 2026-07-03 - Split Image Assets Keeps A Multi-Layer Doc Surface But Consolidates Shared Runtime State
- Decision: Keep the documentation split across canonical docs, runtime-facing skill docs, references, and usage docs, but tighten their authority boundaries and centralize repeated package-state logic in a new `package_state_lib.py`.
- Rationale: The docs had become repetitive, but collapsing them into one or two files would have made the package harder to navigate. The real drift risk was not the number of files by itself; it was unclear reading order plus duplicated implementation logic. `asset_summary` and `plan_manifest` handling were already copied across multiple scripts and were the clearest low-risk refactor targets.
- Impact: `docs/superpowers/split-image-assets/README.md` now acts as a stronger documentation router, `docs/usage/split-image-assets.md` is positioned more clearly as the operator guide, and scripts that mutate/export package state now share `asset_summary` and `plan_manifest` helpers through `package_state_lib.py`.

## 2026-07-03 - Split Image Assets Should Decompose Validator Next, Not Re-merge Docs
- Decision: Treat validator decomposition as the most valuable next architecture milestone after the shared-state consolidation, and do not spend the next round re-merging the documentation surface.
- Rationale: The current package architecture is fundamentally sound. The highest remaining maintenance hotspot is still `validate_asset_package.py`, while the documentation surface is now more navigable thanks to clearer authority routing. Further doc merging would yield less value than breaking down validator domains.
- Impact: Future work should prioritize validator-domain decomposition, then test-module decomposition, and only then revisit deeper generated-route/runtime integration.

## 2026-07-03 - Split Image Assets Validator Now Uses A Thin Entrypoint Plus Domain Modules
- Decision: Replace the monolithic validator implementation with a thin `validate_asset_package.py` entrypoint that orchestrates domain modules for shared helpers, metadata/pipeline validation, object/routing validation, and package-artifact validation.

## 2026-06-25 - Split Image Assets Uses Confirmation Gates For Subjective Split Decisions

Subjective decisions must be asked one at a time with a recommended answer or recorded from prior instructions in `metadata.decision_log[]`, and final `qa.status=pass` should not be claimed for subjective visual decomposition without recorded user acceptance.
